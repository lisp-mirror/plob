Was ist es ungefähr?

Das System PLOB! (Persistent Ljects ISPSOB!) führt orthogonale Ausdauer für LISP- und CLOS-Nachrichten ein. Außer dem Antrag der reinen Ausdauer, auf der einer Seite enthält sie wichtige Datenbankeigenschaften wie Verhandlungen und sperrt und vereinigende Suche über hartnäckigen Nachrichten. Auf der anderen Seite PLOB! ahmt die Eigenschaften des Protokolls Metaobject der CLOS' (REIBEN Sie), soweit wie möglich, z.B. durch seine eingeführten Selbst-self-describing hartnäckigen Nachrichten und Support für die Kategorie Änderungen nach, die dynamisch zu den hartnäckigen Nachrichten gefördert werden (Schemaentwicklung).

Transparente Ausdauer

Ein wichtiges Thema sollte die Schnittstelle zwischen LISP und Ausdauer bilden, die zum Benutzer so transparent ist, wie möglich, ohne Leistungsfähigkeit zu opfern. PLOB! löst beide Probleme: Indem Sie die possibilites von CLOS' einsetzen, REIBEN Sie, die Benutzerschnittstelle zur Ausdauer ist so einfach, wie sie sein kann. Tatsächlich bildet das Verwenden nur einiger Anweisungen über LISP-Code Fälle von den CLOS-Kategorien hartnäckig, z.B.. 
CL-USER 3 > (defclassperson () ((Namens:initarg:name :index (Gleichgestelltes des btree:test)))              
(:metaclass hartnäckig-metaclass))
erklärt eine Kategorie Person mit seinen Fällen, die hartnäckig sind und einem Index für vereinigende Suche auf seinem Schlitz- Namen (wirklich, wird das Kategorie metaobject, das die Kategorie Person selbst darstellt, auch hartnäckig). Das Verwenden einiger anderer Anweisungen bildet die gewöhnlichen hartnäckigen LISP- und Strukturnachrichten. Es ist auch möglich, CLOS-Nachrichten hartnäckig zu bilden, die aus der Steuerung des Benutzers z.B. Fälle heraus sind, in der die defclass, welche die Kategorie definieren, nicht vorhanden ist und nicht geändert werden kann, wie im Beispiel oben angezeigt worden. Außer dem PLOB! hat den Begriff der hartnäckigen Pakete und der hartnäckigen Symbole für das Binden der hartnäckigen Nachrichten: 
CL-USER 4 > (setf #!*aperson * (Bildenfall' Person:name "
Kirschke "))
bindet den hartnäckigen Fall, der vom Bildenfall- Aufruf zum hartnäckigen Symbol festgelegt wird * Einperson *. Ähnlich vorübergehenden Paketen und Symbolen, kann eine Nachricht zu einem hartnäckigen Symbol gesprungen werden und erhält dadurch hartnäckig und identifyable:
CL-USER 5 > (setf #!*azeichenkette * " eine hartnäckige Zeichenkette. ")
... verarbeitet irgendwelche Jahre und LISP später...
CL-USER 99 > #!*azeichenkette *
" eine hartnäckige Zeichenkette. ",

Anwenderarchitektur

PLOB!' s-Architektur ist ein zutreffendes client/server-approach und steht durch UNIX ' StandardRPCs in Verbindung. Der Server übt einen hartnäckigen Nachricht Platz aus; ein oder mehr Klienten können die hartnäckigen Nachrichten zugreifen, die im hartnäckigen Nachricht Platz enthalten werden. Der Gemeinschaftszugriff serialized vollständig, da die eingeführten Verhandlungen und der Sperrungalgorithmus eine Lokalisierung Stufe von 3 entsprechend der Definition SQL-92 ergibt. Der Server ist ein UNIXPROZESSBETRIEB auf einer der Maschinen, die weiter unten erwaehnt werden; die Klienten sind die Prozesse, die eins der LISP-Systeme laufen lassen, die weiter unten erwaehnt werden (besonders, können Klienten mit unterschiedlichen LISP-Systemen an einen Server gleichzeitig anschließen). Indem man UNIX ' StandardRPCs verwendet, wird heterogene Operation auch unterstützt, d.h. können der Server und der Klient auf dem unterschiedlichen Funktionieren oder DIE LISP-Systeme laufen. Zugriff zum hartnäckigen Nachricht Platz kann auf einer Prohauptrechnerstufe erlaubt werden oder verweigert werden.

Leistungsfähigkeit

Leistungsfähigkeit wird durch die niedrigen Schichten gewonnen, die in C eingeführt werden, verbunden mit den hoch entwickelten formenden Daten und der Nachricht Behandlung. Durch das Einführen die meisten niedrigen Arten LISP's direkt als niedrigen Arten in PLOB!, wird die Anforderung für zeitraubende Konvertierungen zwischen vorübergehenden LISP-Nachrichten und hartnäckigen den Nachrichten, die innerhalb des hartnäckigen Nachricht Platzes gespeichert werden, zu einem Minimum gehalten. um den Datenaustausch zwischen Klienten und Server zu verringern, stellt der Klient Code für Nachricht Preallocation und Nachricht das Cachieren zur Verfügung. Eine Referenz von einer ansprechenden hartnäckigen Nachricht auf einer angesprochenen hartnäckigen Nachricht kann als seiend gekennzeichnet werden abhängig; wenn ein Klient den Referenten lädt, wird die angesprochene Nachricht innerhalb des gleichen Datenblockes dem Klienten, auch übermittelt.

Datenbankeigenschaften

Ausdauer selbst ist nur Hälfte des Geschäfts; außer dem PLOB! hat viele Datenbankeigenschaften, nämlich Verhandlungen, Verriegelungen und Indexe eingeführt. Jeder Zugriff zu den hartnäckigen Nachrichten wird in wohlgeformte Verhandlungen, irgendeine durch einen ausdrücklichen Verhandlungblock eingebettet, der durch den Code des Benutzers geöffnet ist:
CL-USER 6 > (Mitverhandlung ()
             (setf #!*azeichenkette * " folgende hartnäckige Zeichenkette. ")
              ... Lots mehr Codes...
             (setf #!*bzeichenkette * " zweite Zeichenkette. "))
oder, wenn der Benutzer nicht einen Verhandlungblock spezifizierte, PLOB! umgibt jeden Zugriff durch eine Verhandlung innerlich. **time-out** in d oben genannt Beispiel, einbetten d Code in ein Verhandlung versichern daß entweder all ändern tun zu hartnäckig Nachricht in d Block werden sein festlegen zu d hartnäckig Nachricht Platz oder, z.B. in d Schachtel von ein Fehler an fortfahren d Anweisung in d Block, kein von d Änderung werden sein fördern zu d Datenbank. Periodische Veröffentlichung zwischen gleichzeitigen Verhandlungen wird durch das Zweiphasenverhandlungmodell zur Verfügung gestellt, das innerhalb PLOB angenommen wird!

Die Vorbedingung, damit eine Verhandlung ist wohlgeformt ist, daß jede Nachricht, die innerhalb einer Verhandlung erreicht wird, für den Accessor reserviert ist; dieses wird durch hierarchisches ProtokollVerriegelung PLOB ' s vollendet. Verriegelungen können auf einer des Stufen Schlitzes, Nachricht angefordert werden, oder die vollständige Datenbank mit den Modi gelesen worden, schreiben oder Read-only. Das Seite Sperrung wird nicht unterstützt, da dieses kein sinnvoll für die heterogene Bevölkerung ist, die in einer objektorientierten Datenbank gefunden wird.

Für Indexe PLOB! hat hartnäckige btrees, die innerlich für verschiedene Tabellen verwendet werden und für den Zweck des Benutzers zur Verfügung gestellt werden, auch wie im Person- Kategorie Beispiel bereits oben gezeigt. Es ist auch möglich, btrees direkt zu verwenden: 

CL-USER 7 > (mein-btree setf * * (bildenSie:test-'
Gleichgestelltes))
stellt * mein-btree * auf ein hartnäckiges btree ein. Die Operationen auf hartnäckigen btrees sind denen der Hash-Tabellen, mit dem Unterschied der Tasten sehr ähnlich, die sortiert werden. (selbstverständlich, werden LISP's-Hash-Tabellen auch unterstützt; PLOB! verletzt kein Diagramm von Hash-Tabellen zu btrees.),

PLOB! wird als shareware in der GNU-Öffentlichkeit Lizenz ausgedrückt, sehen die Anleitung des Benutzers für Details verteilt. PLOB! läuft auf folgenden LISP und Betriebssysteme:

Unterstützt allgemeine LISP- Version 3,2 Harlequin LispWorks und allgemeine LISP- Version 4,3 und 5,0 Franz Allegro Beta. Für Windows/NT wird allgemeine LISP- Version 4,0,1 Harlequin LispWorks unterstützt. Ist allgemeine LISP- Version 4,3,2 Franz Allegro für Windows/NT nicht überprüft worden.

Eine Klient-nur Anlage sollte für andere Betriebssysteme mit der Verteilung möglich sein, die hier bereitgestellt wird, aber benötigt eine Maschine Solaris, IRIX, Linux oder Windows/NT, die den PLOB- Server nahe dem Klienten laufen läßt.

Unterschiedlicher Verbrauch von Verhandlungen und von Sperrung

In diesem Kapitel wird die Leistung für die einzelne Operation von Jobstep 9,aber mit unterschiedlichem Verbrauch von Verhandlungen und von Nachricht Sperrung gezeigt. Das Rückstellung Verhalten durch PLOB, wie in Jobstep 9 gezeigt bezieht eine Verhandlung für jeden Aufruf Bildenfall und den (impliziten) Verbrauch einiger Verriegelungen an der Speicherung der Nachricht mit ein. Das Verwenden von Verhandlungen und von Verriegelungen auf grösseren `chunks der Nachrichten verbessert Leistung für einen einzelnen Benutzer erheblich, aber wird das `granularity ' vom Nachricht Zugriff gröber bilden und bildet gleichzeitigen Zugriff langsamer.
